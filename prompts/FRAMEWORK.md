# Prompt 框架总览

> 参考 inkos 内容生产体系 × YC Office Hours 六问法 × 剧本杀行业最佳实践

---

## 框架设计原则

### 1. 先定义，再生成

```
不先问"怎么写"，而是先问"写什么"和"为什么写"。
每个创作阶段都有前置条件，不满足不进入下一步。
```

### 2. 信息差即产品

```
剧本杀不是小说，是信息差的艺术。
每个角色只知道局部真相，所有信息块拼在一起才是完整真相。
框架的每个环节都围绕"信息边界"设计。
```

### 3. 约束即质量

```
prompt 不是越详细越好，而是约束越清晰越好。
每个 prompt 包含：必须包含 / 不能包含 / 输出格式。
模糊的 prompt = 模糊的输出。
```

## Layer 0 — 创意问卷

**文件：** `00_guided_questionnaire.md`

**作用：** 创作前的创意碰撞，类似 YC Office Hours 六问法

**核心问题：**
1. 题材定位 — 什么类型的剧本杀？核心瞬间是什么？
2. 核心冲突 — X vs Y = ? 的方程式
3. 人物约束 — 为什么这6个人必须被困在一起？
4. 情感设计 — 玩家记住的3个情绪时刻
5. 差异化 — 和同类剧本比独特在哪？
6. 上线标准 — 怎么判断"做完了"？

**输出：** `brief.md` — 所有后续 prompt 的默认上下文

---

## Layer 1 — 机制设计

**文件：** `01_mechanism_design.md`

**顺序：** Q1概念验证 → Q2原型设计 → Q3平衡验证 → Q4扩展包

**关键原则：**
- 机制服务于核心方程式，不是为特色而特色
- 平衡性验证不是可选项，是强制步骤
- 每个机制必须回答：玩家做什么操作，代价是什么？

---

## Layer 2 — 剧本创作

**文件：** `02_script_generation.md`

**顺序：Q1信息矩阵 → Q2角色背景 → Q3a本 → Q4b本 → Q5DM手册 → Q6结局**

**核心原则：**
- 信息矩阵先于任何角色剧本存在
- 每个角色有"可以说 / 不能主动说 / 绝对否认"三约束
- b本不是a本续集，是"角色被迫面对新信息"的阶段

---

## Layer 3 — 穿帮检测

**文件：** `04_consistency_check.md`

**三轮审计：**
1. 信息矩阵逐格检查（角色 × 事件，每格必查）
2. 机制一致性检查（角色行为是否符合机制设定）
3. 结局合理性检查（触发条件是否可推导，结局是否有唯一解）

**P0 问题：** 必须修复才能上线
**P1 问题：** 建议修复，影响体验
**P2 问题：** 可选优化

---

## Layer 4 — 商业化

**文件：** `05_commercialization.md`

**内容：** 成本核算 / 定价策略 / 渠道分析 / 利润估算

## Prompt 编写规范

### 标准结构

```markdown
## [环节名称]

### 输入
从哪读取什么上下文（文件路径或直接引用）

### 约束
- 必须包含：（3条以内）
- 不能包含：（3条以内）

### 输出格式
明确的格式要求，最好有示例结构

### 自检清单
完成后 AI 自我检查的条目
```

### 约束强度标记

| 标记 | 含义 |
|------|------|
| **[必]** | 不满足则整个输出无效 |
| **[建]** | 建议遵守，不强制 |
| **[示]** | 示例，可替换 |
| **[禁]** | 绝对禁止出现 |
