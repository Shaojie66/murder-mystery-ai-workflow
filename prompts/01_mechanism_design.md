# 机制设计 Prompt

> 适用于 Gemini 3 Pro / Claude 深度研究模式

---

## 使用说明

每个 prompt 包含三个部分：
- **输入**：AI 需要知道的上下文（从 brief.md 和上游产物读取）
- **约束**：必须遵守的规则
- **输出**：明确的格式要求

**顺序**：Q1（概念验证）→ Q2（原型设计）→ Q3（平衡验证）→ Q4（扩展包）

---

## Q1：概念验证

```
【brief.md 核心方程式】

【可选：参考规则书内容】

你是一个严苛的机制评审员。在设计任何具体规则之前，先回答：

1. 这个"核心方程式"有**可玩性**吗？
   - 两个对立价值是否都能在游戏中被**真实体验**到？
   - 玩家是否需要做**真正的权衡**而非走流程？

2. 这个类型的市场验证：
   - 同类成功剧本有哪些？（至少3个）
   - 它们的机制共同点是什么？
   - 你的差异化是否真的有意义，还是自嗨？

3. 最危险的假设是什么？
   - 如果这个假设是错的，整个设计会怎样？

请给出：
- 概念评分（1-10，附理由）
- 如果 < 7，说明哪里需要重构
- 3个以内**必须解决**的核心问题

格式：Markdown，评审结构清晰
```

## Q2：原型机制设计

```
基于以下已验证的概念，设计游戏机制：

【brief.md 完整内容】
【Q1 评审结果 + 修正方向】

## 必须包含

### 2.1 核心机制（1-2个，不超过3个）
每个机制需说明：
- 名称 + 一句话描述
- 玩家**做什么操作**
- 操作的**代价/限制**是什么
- 这个机制**如何服务核心方程式**

### 2.2 游戏阶段
- 总阶段数：3-5个（含开场/中盘/结局）
- 每个阶段的**主要活动**是什么
- 阶段之间的**信息流动**是怎样的

### 2.3 阵营/角色分工
- 是否有阵营？有的话几个？
- 如果没有阵营，6人角色各**承担什么功能**（不是背景，是游戏功能）

### 2.4 胜负条件
- 胜者如何获胜？给出**具体判定条件**
- 是否有**多解**？还是只有一个真相？
- 凶手获胜的概率是否接近50%？

### 2.5 搜证/信息获取机制
- 玩家如何获得信息？（投票/搜证/技能/交换）
- 信息获取是否有**随机性**？如何控制运气成分？
- 是否有"死路"（无论如何选择都必输）的风险？

### 2.6 投票/淘汰机制
- 投票时机是什么？
- 是否有**连续淘汰**？如果有，机制如何防止一边倒？
- 弃票/平票如何处理？

### 2.7 特色机制（可选）
- 描述你最有记忆点的那个设计
- 它是**必要**的吗？还是为了特色而特色？

## 输出格式
```markdown
## 概念评分延续
（Q1分数 → 现在的分数，说明改进了什么）

## 核心机制
（每个机制一节，说明上述5点）

## 游戏流程
（时间轴：开场X分钟 → 中盘Y分钟 → 结局Z分钟）

## 角色分工矩阵
| 角色 | 游戏功能 | 信息优势 | 生存能力 |
|------|----------|----------|----------|
（6行，对应6个玩家位）

## 胜负判定树
（用流程图或伪代码描述判定逻辑）

## 潜在问题
（列出3个你最没把握的地方）
```
```

## Q3：平衡性验证

```
请验证以下机制的平衡性：

【完整机制设计】

验证维度：

### 角色强度分析
- 每个角色的**信息获取能力**是否接近？（差距不能超过20%）
- 每个角色的**存活能力**是否有差异？（某些角色是否天生更容易被集火？）
- 每个角色的**翻盘能力**如何？（落后时是否有逆转手段？）

### 信息差验证
- 凶手是否比好人有**合理的信息优势**？
- 好人的信息获取路径是否**足够多样**？（不依赖单一线索）
- 是否存在"正确答案路径"——只有一种方式能破局？

### 时间节奏验证
- 前30分钟玩家是否**有事可做**？（不能只是读本）
- 中盘是否有**信息爆炸**？（线索太多导致分析瘫痪）
- 结局前是否有**关键抉择点**？（让玩家感觉自己的选择有意义）

### 漏洞扫描
- 是否存在"必定胜利"的策略组合？
- 是否有角色在任何情况下都**无法获胜**？
- 凶手是否可以通过"不说话"来稳定获胜？

对于每个问题，给出：
- 问题描述
- 严重程度（P1/P2/P3）
- 具体修复建议

## 输出格式
```markdown
## 平衡性评分
总评分 X/10
| 维度 | 评分 | 主要问题 |
|------|------|----------|

## 问题清单
| ID | 维度 | 严重程度 | 描述 | 修复建议 |
|----|------|----------|------|----------|
| 1 | | P1 | | |

## 调整方案
（按优先级列出具体的数值调整或规则修改）
```
```

## Q4：扩展包概念

```
基于以下已完成的机制，设计2-3个扩展包方向：

【完整机制 + 平衡性报告】

每个扩展包需包含：
- **名称** + 一句话定位
- **新增机制**（不要重复核心设计的机制）
- **适合人群**（新手/硬核/情感向）
- **与核心的冲突点**（扩展包是否会破坏核心体验？）

可选方向参考：
- 时间维度（多周目/时间循环）
- 空间维度（新增地图区域）
- 阵营维度（新增阵营/拆分原有阵营）
- 角色维度（新增角色/替换现有角色）

格式：每个扩展包一节，500字以内
```
