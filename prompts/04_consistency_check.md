# 穿帮检测 Prompt

> 适用于 Claude 深度审计

---

## 三轮审计体系

每轮审计独立运行，各自输出专项报告，最后由 LLM 综合评分。

---

## 第一轮：信息矩阵逐格检查

```
【6位角色a本 + b本（共12份）】
【信息矩阵（带认知状态标注）】

## 任务
对照信息矩阵，逐角色、逐事件检查剧本描述是否与矩阵一致。

## 检查维度

### 维度1：知识边界
角色是否在某个事件中提及了当时还不知道的信息？
- 矩阵标注[否]的信息被提及 → P0
- 矩阵标注[疑]的信息被当作[知]使用 → P1

### 维度2：时间线
角色对事件的描述顺序是否与矩阵时间线一致？
- 同一事件内的时间顺序矛盾 → P1
- 跨事件顺序颠倒（先知道后发生的事）→ P0

### 维度3：秘密泄露
角色的秘密是否被其他角色意外知道（矩阵中不该存在知识流动）？
- 角色A的秘密被角色B在a本中提及 → P0
- 角色A的秘密被角色B通过b本搜证获得 → P1（如果是预期内的）

### 维度4：视角真实性
每个角色的"真相"是否与其矩阵中的角色设定一致？
- 角色行为与背景设定矛盾 → P1
- 角色心理描写与已知信息不匹配 → P2

## 输出格式
```markdown
## 第一轮审计：信息矩阵逐格

### 总体评分
P0:X个 | P1:X个 | P2:X个
上线评估：[可上线/需修改/不建议上线]

### 逐角色问题清单
#### 角色R1：{角色名}
| 问题ID | 维度 | 严重程度 | 事件 | 问题描述 | 涉及角色 | 修复建议 |
|--------|------|----------|------|----------|----------|----------|
```

## 第二轮：机制一致性检查

```
【机制设计.md】
【完整6人角色剧本】
【信息矩阵】

## 任务
检查角色剧本中的行为是否与机制设计一致。

## 检查维度

### 维度1：阵营一致性
- 阵营设定是否自洽？（同阵营角色是否都按阵营利益行动）
- 阵营技能是否与角色背景匹配？（没有凭空冒出的能力）

### 维度2：技能/道具可达性
- 角色剧本中提及的技能/道具，在机制中是否有定义？
- 关键道具的获取路径是否在剧本中有铺垫？

### 维度3：胜利路径
- 每个角色的胜利条件是否在剧本中有对应情节支撑？
- 角色是否有不止一条胜利路径？（如果只有一条，是否合理）

### 维度4：死路检测
- 是否存在角色"无论如何选择都必输"的情况？（机制漏洞）
- 是否存在"无论发生什么凶手必定获胜"的情况？（平衡性P0）

### 维度5：信息获取节奏
- 玩家在游戏前中期是否有足够信息支撑推理？
- 信息是否过度集中于某个阶段（导致前期无聊/后期爆炸）？

## 输出格式
```markdown
## 第二轮审计：机制一致性

### 总体评分
P0:X个 | P1:X个 | P2:X个
上线评估：[可上线/需修改/不建议上线]

### 问题清单
| ID | 维度 | 严重程度 | 问题描述 | 修复建议 |
|----|------|----------|----------|----------|
```

## 第三轮：结局合理性检查

```
【完整6人剧本 + DM手册】
【信息矩阵】
【机制设计.md 胜负条件】

## 任务
验证结局设计是否合理、是否有唯一解。

## 检查维度

### 维度1：结局唯一性
- 结局是否只有一个正确答案？还是存在多个合理解释？
- 如果是多结局，不同结局的触发条件是否清晰可判断？

### 维度2：触发条件可推导性
- 触发结局所需的条件，是否在游戏过程中对玩家可见/可推理？
- 还是只能靠DM陈述（被动触发）？

### 维度3：角色参与感
- 是否有角色在结局时"消失"（全程无戏份）？
- 每个角色的选择是否对结局有真实影响？

### 维度4：情感峰值
- 结局是否给玩家留下深刻印象？
- 是否有"记忆点"（事后玩家会向朋友描述的瞬间）？

### 维度5：复盘价值
- 结局是否值得复盘？（玩家是否想再玩一次探究不同选择的结果）
- DM复盘时是否有话可说？（不是"你们运气不好"，而是有具体分析）

## 输出格式
```markdown
## 第三轮审计：结局合理性

### 总体评分
P0:X个 | P1:X个 | P2:X个
上线评估：[可上线/需修改/不建议上线]

### 问题清单
| ID | 维度 | 严重程度 | 问题描述 | 修复建议 |
|----|------|----------|----------|----------|

### 记忆点评估
- 有记忆点的场景：X个
- 建议突出：{描述最有力的1-2个场景}
```

## 综合评估

将三轮审计结果汇总，生成最终报告：

```markdown

> 审计时间：{日期}
> 审计版本：{commit hash}

## 问题统计
| 轮次 | P0 | P1 | P2 |
|------|----|----|-----|
| 第一轮 | | | |
| 第二轮 | | | |
| 第三轮 | | | |
| **总计** | | | |

## 上线评估
**[可上线 / 需修改后上线 / 不建议上线]**

## 必须修复（P0）
| # | 位置 | 问题 | 修复方式 |
|---|------|------|----------|

## 建议修复（P1）
| # | 位置 | 问题 | 修复方式 |

## 可选优化（P2）
| # | 位置 | 问题 | 修复方式 |

## 修复优先级
1. {最紧急的修复}
2. {次要修复}
3. {可选优化}
```
