## 1. 摘要

murder-wizard Pro 是 murder-wizard CLI 工具的在线升级版本，以订阅制 SaaS 形式交付。核心价值：云端存储、团队协作、AI 穿帮检测、高级模板、优先 LLM 通道。目标用户是中国剧本杀创作者（业余爱好者和职业创作者），定价 ¥29/月。CLI 版本继续免费提供，Pro 版本面向需要协作、云端存储和 AI 辅助的专业创作者。

## 3. 背景

### 3.1 现状

murder-wizard 是一款 Python CLI 工具，提供 8 阶段剧本杀创作工作流（机制设计 → 剧本创作 → 视觉物料 → 用户测试 → 商业化 → 印刷生产 → 宣发内容 → 社区运营）。用户通过终端命令交互，产物本地存储。

已有种子用户，主要为技术背景的剧本杀爱好者。

### 3.2 为什么现在要做

剧本杀中国市场规模持续扩大（2025 年预估 ¥50 亿+），创作者需求分层明显：

- **业余创作者**：创意热情高，技术背景弱，CLI 上手门槛高
- **职业创作者/工作室**：有多人协作、版本管理、云端同步的需求
- **发行商**：需要快速生产、批量管理多个剧本项目

现有 CLI 只能服务第一类用户，且数据在本地无法跨设备访问。Web 化是扩大用户基数和商业化的必经之路。

### 3.3 机会窗口

- 中国剧本杀内容需求旺盛，创作者愿意为效率工具付费
- AI LLM 技术成熟，AI 辅助创作从"噱头"变为"实用工具"
- 市场上尚无成熟的剧本杀创作 SaaS，竞品空白

## 6. 价值主张

### 6.1 核心价值

**为剧本杀创作者提供的核心价值**：

1. **省时间**：AI 辅助创作，自动生成角色、线索、剧本，比手工快 5-10 倍
2. **不穿帮**：AI 穿帮检测，自动发现角色间信息矛盾，避免玩家体验破裂
3. **随时访问**：云端存储，在任何设备上打开浏览器就能继续创作
4. **团队协作**：多人同时编辑，角色分配、评论、版本历史一目了然

## 7. 解决方案

### 7.1 产品路线图

```
第一版（MVP）─────────────── 第二版 ──────────────── 第三版
  │                              │                      │
  ├─ 云端项目存储               ├─ 团队协作（实时编辑）  ├─ 发行商对接
  ├─ 引导式 Web 界面           ├─ 角色分配 + 权限管理  ├─ 批量项目管理
  ├─ AI 穿帮检测（基础版）     ├─ 高级模板市场         ├─ API 开放
  ├─ 基础模板库                ├─ AI 穿帮检测（增强版） │
  └─ 订阅支付                   └─ 消息通知系统         │
```

### 7.2 第一版功能详情（MVP）

#### F1：用户系统

- **注册/登录**：邮箱 + 密码（后续支持微信）
- **个人设置**：修改密码、订阅管理、头像

#### F2：云端项目存储

- **创建项目**：输入剧本名称、类型（原型/完整）
- **项目列表**：展示所有项目，显示名称、创建时间、最后更新时间
- **云端同步**：本地 CLI 项目可绑定云端账号（通过 `murder-wizard login` 关联）

#### F3：引导式 Web 界面

- **新建剧本向导**：类似 Notion block 的分步编辑，引导用户完成 8 阶段
- **阶段仪表盘**：每个阶段显示完成状态、产物预览、一键继续
- **响应式设计**：支持桌面和移动端（手机可查看/编辑）

#### F4：AI 穿帮检测（基础版）

- **自动扫描**：在阶段 2（剧本创作）完成后，自动运行检测
- **问题列表**：列出发现的信息矛盾，标注角色、事件位置
- **修复建议**：对每个问题给出修改建议

#### F5：订阅支付

- **定价**：¥29/月，按月计费
- **支付方式**：支付宝、微信支付（国内）
- **订阅管理**：随时取消、重新订阅、发票申请

### 7.3 关键用户流程

#### 流程 1：首次注册到完成第一个剧本

```
1. 访问 landing page → 点击"立即开始"
2. 注册（邮箱 + 密码）
3. 选择套餐（免费试用 7 天 / 直接订阅）
4. 创建第一个项目（输入剧本名称）
5. 进入引导式创建流程（阶段 1：机制设计）
6. 按向导提示逐步完成各阶段
7. 完成阶段 2 后，AI 自动运行穿帮检测
8. 查看检测结果，修复问题
9. 进入阶段 3-6，完成视觉物料、测试、印刷准备
10. 导出 PDF
```

#### 流程 2：团队协作（第二版）

```
1. 项目创建者打开"团队设置"
2. 输入协作者邮箱，发送邀请
3. 协作者接受邀请，获得项目访问权限
4. 协作者可以看到实时编辑状态
5. 创建者可以分配角色（谁负责哪个角色剧本）
6. 所有人的编辑自动保存到云端
7. 可以查看版本历史，回滚到任意版本
```

### 7.4 技术架构（概要）

| 层级 | 技术方案 |
|------|---------|
| 前端 | React + TypeScript + Tailwind CSS |
| 后端 | Python FastAPI（复用 CLI 核心逻辑） |
| 数据库 | PostgreSQL（用户、项目、订阅） |
| 文件存储 | S3 兼容存储（MinIO / 阿里云 OSS） |
| LLM 集成 | Claude API / GPT-4o / Ollama（与 CLI 一致） |
| 支付 | stripe（国际）+ 支付宝/微信（国内，通过第三方） |
| 部署 | Docker容器，K8s 或云函数 |

### 7.5 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 前端框架 | React + Tailwind | 团队已有技术栈，复用 murder_wizard_web |
| 引导式 vs 自由编辑 | 引导式为主，可切换"专家模式" | 降低新手门槛，高级用户可跳过引导 |
| 协作算法 | CRDT（Yjs）或 OT | 支持多人实时编辑 |
| 穿帮检测精度 | 保守策略（低误报） | 初期宁可漏报，不要误报破坏信任 |
| 订阅层级 | 单一 Pro 订阅（¥29/月） | 简化决策，后续可扩展团队版 |

### 7.6 假设与风险

| 假设 | 风险 | 缓解措施 |
|------|------|---------|
| 用户愿意为云端功能付费 | 市场验证不足 | 先做 Fake Door 实验，收集邮箱验证意愿 |
| AI 穿帮检测准确率 > 85% | 技术实现难度高 | 先做技术验证，确认准确率后再作为卖点 |
| 引导式 UI 能兼顾新手和高级用户 | 两类需求冲突 | 设计"专家模式"开关，允许高级用户跳过引导 |
| 国内用户接受在线订阅支付 | 支付习惯差异 | 接入支付宝/微信，参考国内订阅产品定价策略 |

### 8.2 第一版范围（MVP）

- 用户注册/登录
- 云端项目存储（无限项目）
- 引导式 8 阶段工作流（Web 版）

- 基础模板库（≥ 5 个模板）

**不在范围内（第二版）**：
- 实时团队协作（多人同时编辑）
- 高级模板市场（用户上传/买卖模板）
- 发行商对接
- API 开放
